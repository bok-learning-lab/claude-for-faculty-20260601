It's all text What goes in, what comes out, and where you keep it

A model only ever reads text. Whatever you hand it — a PDF, a spreadsheet, a web page — is first flattened to plain text, chopped into tokens, turned into numbers, and read in order. It writes tokens back, in order. Text in, text out.

A token is a chunk between a letter and a word. The model never sees your file — only this one line of tokens, in sequence.

Where it lives: text files

free-form structured
.txtPlain text No formatting at all — just characters. lecture transcripts, oral-history interviews, primary sources, field notes, Shakespeare's sonnets
.rtfRich text A little formatting: bold, italics, headings. handouts, recommendation letters, annotated readings
.mdMarkdown Light formatting from a few simple marks. The lingua franca of this work. syllabi, lecture notes, reading guides, problem sets, rubrics, prompts & skills, CLAUDE.md
.htmlWeb markup Tagged structure a browser can render. course sites, online readings, digital exhibits — like this handout
.csvComma-separated values Rows and columns — a table in plain text. gradebooks, survey results, coded interview data, corpus & archive metadata, lab measurements
.jsonStructured data Labelled key / value pairs. Strict rules. citation records, structured annotations, dataset & config files, API responses
.py .jsCode Scripts that actually run. The strictest rules of all. text-mining & NLP, data cleaning, statistics, visualizations, grading automations

One session = one context window

200K – 1M tokens  →
room left →
system prompt memory your first prompt inputs instructions / operations output

It's all tokens in one window. It starts partly full — a small system prompt and memory — then each turn adds your prompt, your inputs (usually the biggest piece), the operations, and the output. New inputs often come from earlier outputs, so it loops and accumulates until the window fills.
